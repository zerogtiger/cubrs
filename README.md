# Cubrs

A fast Rubik's cube solver implemented in Rust using Kociemba's two-phase algorithm, with a terminal-based speedcubing timer.

## Usage

### As a Speedcubing Timer

Run the binary for an interactive terminal timer:

```
cargo run --release
```

**Controls:**
- **Space** — press once to arm, then again to start the timer
- **Any key** — stop the timer
- **q** — quit

The timer displays the scramble and a visual cube state before each solve.

Note that tables required for the algorithm are generated automatically on first run and stored in 
a `tables/` directory for ~260 mb. This may take a few minutes if running for the first time. 

### Cubie API

The `Cubie` struct is the core cube representation.

#### Construction

```rust
// Solved cube
let cube = Cubie::default();

// From explicit arrays
let cube = Cubie::new(corner_permutation, corner_orientation, edge_permutation, edge_orientation);

// From a 54-character facelet string (URFDLB order, center-determined colors)
let cube = Cubie::from_string("UUUUUUUUURRRRRRRRRFFFFFFFFFDDDDDDDDDLLLLLLLLLBBBBBBBBB").unwrap();

// From a Facelet representation
let cube = Cubie::from_facelet(&facelet).unwrap();
```

Note that the string specified in the `from_string` function encodes the colors on each side in the 
order: $U_0 \dots U_8\; R_0 \dots R_8\; F_0 \dots F_8\; D_0\dots D_9\; L_0 \dots L_8\; B_0 \dots 
B_8$. Whitespaces will be ignored. 
```
         .--------.
         |U0 U1 U2|
         |U3 U4 U5|
         |U6 U7 U8|
.--------:--------:--------.--------.
|L0 L1 L2|F0 F1 F2|R0 R1 R2|B0 B1 B2|
|L3 L4 L5|F3 F4 F5|R3 R4 R5|B3 B4 B5|
|L6 L7 L8|F6 F7 F8|R6 R7 R8|B6 B7 B8|
'--------:--------:--------'--------'
         |D0 D1 D2|
         |D3 D4 D5|
         |D6 D7 D8|
         '--------'
```

Note that you can use any character to specify the color, as long as it is consistent across the 
specification. The color for each side is auto-detected via the center tile. For instance
```rust
let mut cube = Cubie::from_string(
    "BGGBWWRBB WBWGROGYO BYOYGWGBY WRROYGRYY OOYROOBGR ORYWBWGRW",
).expect("Invalid specification");
```

#### Applying Moves

```rust
// Apply a single move, returns a new Cubie (non-mutating)
let cube = cube.apply_move(Move::U);

// Apply a sequence of moves
let cube = cube.apply_moves(&[Move::R, Move::U, Move::R3, Move::U3]);
let cube = cube.apply_moves(&Move::move_list_from_str("R U R' U'").unwrap());
```

#### Validity Checking

```rust
// Returns true if the cube state is physically reachable from the solved state
cube.is_solvable() // bool
```

#### Multiplication

`Cubie` implements `Mul` for composing two cube states. `A * B` means "apply B first, then A":

```rust
let composed = cube_a * cube_b;
```

This follows the standard cubie composition rule:
- `(A*B)(x).corner = A(B(x).corner).corner`
- `(A*B)(x).orientation = A(B(x).corner).orientation + B(x).orientation`

### CubeDisplay

`CubeDisplay` renders a cube state to the terminal as a colored ASCII cross-shaped net (U on top, 
L/F/R/B in the middle row, D on the bottom).

```rust
// Default colors: white, red, green, yellow, orange, blue with ANSI background colors
let display = CubeDisplay::default_colors();

// Or specify your own
let display = CubeDisplay::new(
    [
        "\x1b[48;5;15m\x1b[38;5;0mW \x1b[0m",  // U = white
        "\x1b[48;5;9m\x1b[38;5;0mR \x1b[0m",   // R = red
        "\x1b[48;5;10m\x1b[38;5;0mG \x1b[0m",  // F = green
        "\x1b[48;5;11m\x1b[38;5;0mY \x1b[0m",  // D = yellow
        "\x1b[48;5;208m\x1b[38;5;0mO \x1b[0m", // L = orange
        "\x1b[48;5;32m\x1b[38;5;0mB \x1b[0m",  // B = blue
    ],
    2, // width of each color cell in characters
);
```

### Solver

```rust
let solver = Solver::new();

// Solve a cube state
let solution = solver.solve(&cube, 21).unwrap();
println!("{solution:?}");

// Generate a random scramble
let (cube, scramble) = solver.generate_scramble(21);
```

Note the scramble is generated by randomly selecting a reachable cube state, then inverting the 
generated solution to ensure bias free scramble generation. 

## Algorithm details

This project implements Herbert Kociemba's two-phase algorithm to optimally (or near-optimally) able to solve any Rubik's cube state in at most 20 moves. It includes:

- A full implementation of the two-phase IDA* search
- Symmetry-reduced pruning tables for fast search
- Disk-based table persistence to avoid regeneration on startup
- A terminal UI for use as a speedcubing practice timer

### Phase 1

Phase 1 searches for a sequence of moves that brings the cube into the subgroup G1 = ⟨U, D, L2, R2, F2, B2⟩. A cube is in G1 when:

- All edges are correctly oriented (edge orientation coordinate = 0)
- The 4 UD-slice edges (FR, FL, BR, BL) are in the UD-slice positions

Phase 1 uses a pruning table indexed by *(symmetry-reduced FlipUDSlice × corner orientation)*, with 64,430 × 2,187 ≈ 140M entries. The FlipUDSlice coordinate is symmetry-reduced using the 16-element subgroup of cube symmetries that fix the UD axis, reducing the search space by up to 16×.

### Phase 2

Phase 2 searches for a sequence of phase-2-only moves that fully solves the cube from the G1 state found in phase 1. Only moves from {U, U2, U', D, D2, D', R2, F2, L2, B2} are used, which preserve G1.

Phase 2 uses a pruning table indexed by *(symmetry-reduced corner permutation × non-slice edge permutation)*, with 2,768 × 40,320 ≈ 111M entries. The UD-slice permutation (24 states) is not tracked during the search — instead it is verified at leaf nodes.

### IDA* Search

The two phases are interleaved in a single IDA* search:

- Phase 1 runs with increasing depth limits
- Every time a G1 state is reached at exactly the current depth limit, phase 2 is immediately attempted with the remaining budget
- When a solution is found, the depth limit is tightened to search for shorter solutions

## Pruning Tables

Both pruning tables use symmetry reduction and require special handling for **symmetric positions** — cube states whose symmetry class has self-symmetries beyond the identity. For such states, multiple table entries correspond to the same physical state and must all be set simultaneously during table generation to avoid incorrect depths.

Tables are generated once and saved to disk in binary format for fast subsequent loading.

| Table | Size | Description |
|---|---|---|
| `phase_1_prune.bin` | ~140 MB | Phase 1 pruning: FlipUDSlice sym × corner orient |
| `phase_2_prune.bin` | ~111 MB | Phase 2 pruning: corner perm sym × edge perm |
| `move_flip_ud_slice.bin` | ~4.6 MB | FlipUDSlice sym-coord move table |
| `move_corner_orient.bin` | ~0.07 MB | Corner orientation move table |
| `move_corner_perm_sym.bin` | ~0.3 MB | Corner perm sym-coord move table |
| `move_phase2_edge_perm.bin` | ~2.9 MB | Phase 2 edge perm move table |
| `move_phase2_ud_slice.bin` | ~0.003 MB | UD slice move table |
| `twist_conjugate.bin` | ~0.07 MB | Corner orientation conjugation table |
| `edge8_pos_conjugate.bin` | ~1.2 MB | Phase 2 edge perm conjugation table |

Table generation takes several minutes on first run. Subsequent runs load from disk.


## Coordinate System

The solver uses the following coordinates to represent cube states. However, only the corner 
orientation/permutation and edge orientation/permutation is sufficient to uniquely determine a cube 
state. The rest are used for solver optimization purposes.

| Coordinate | Range | Description |
|---|---|---|
| Corner orientation | 0–2186 | How each corner is twisted ($3^7$) |
| Edge orientation | 0–2047 | How each edge is flipped ($2^{11}$) |
| UD slice | 0–494 | Which 4 positions contain UD-slice edges |
| FlipUDSlice (sym) | 0–64429 | Combined edge orient + UD slice, symmetry reduced |
| Corner permutation | 0–40319 | Arrangement of 8 corners (8!) |
| Corner perm (sym) | 0–2767 | Corner permutation, symmetry reduced |
| Phase 2 edge perm | 0–40319 | Arrangement of 8 non-slice edges (8!) |
| Phase 2 UD slice | 0–23 | Arrangement of 4 UD-slice edges (4!) |

## Building

```
cargo build --release
```

Requires Rust 1.70+. Tables are generated automatically on first run and stored in a `tables/` directory.
